# AGENTS Guide

This file explains how to work safely and effectively in this repository, especially when changes are being proposed or generated by AI coding tools.

## Purpose

This repository contains a Terraform provider for Cassandra schema migration workflows.

The provider intentionally separates:

- user-level schema ownership
- system-level operational ownership

That separation is a core design constraint, not just a documentation preference.

## Project goals

- Let application teams manage table shape safely.
- Let platform or database teams manage operational settings safely.
- Avoid accidental destructive behavior.
- Keep Cassandra schema evolution explicit, reviewable, and predictable.

## Repository layout

- `main.go`: provider entrypoint
- `internal/provider`: provider, client, and resource implementations
- `examples`: reference Terraform configurations
- `README.md`: public project overview and usage guidance
- `CONTRIBUTING.md`: contributor workflow and expectations

## Non-negotiable design rules

- Do not blur ownership boundaries between user-level and system-level resources.
- Do not introduce implicit destructive behavior.
- Do not silently change immutable schema concepts such as partition keys or clustering keys.
- Do not hide migration risk behind convenience APIs.
- Prefer explicit validation and clear error messages over clever or ambiguous behavior.

## AI coding best practices

When using AI to assist with changes in this repository:

- Read existing code before editing so changes match the current provider model.
- Start each distinct feature or task on its own branch.
- Prefer using a dedicated git worktree under `.worktree/` for each separate feature or task so parallel changes stay isolated.
- Make the smallest safe change that solves the problem.
- Preserve public behavior unless the change explicitly intends to alter it.
- Keep generated code easy for humans to review and maintain.
- Prefer clarity over abstraction; avoid introducing extra layers unless they materially improve the design.
- Follow the established naming and resource boundaries already present in `internal/provider`.
- Update documentation and examples whenever behavior or workflow changes.
- Treat Terraform resource semantics as part of the public API.

## Safe change rules for AI and humans

- Never add code that drops tables or indexes automatically.
- Never assume Cassandra supports unsafe in-place type migrations.
- Never introduce state behavior that could surprise Terraform users during plan or apply.
- Never edit unrelated files just because they are nearby.
- Never overwrite or revert user changes that are already in progress.

## Code quality guidance

- Keep functions focused and readable.
- Prefer explicit error handling with actionable messages.
- Reuse existing patterns in the provider unless there is a clear reason to improve them.
- Add comments only where intent is non-obvious.
- Keep docs aligned with implementation.

## Testing expectations

Before merging changes, run:

```bash
gofmt -w .
go vet ./...
go test ./...
```

If behavior changes and automated coverage is not practical yet:

- explain what was validated manually
- document any remaining risks in the pull request

## Documentation expectations

Update these when relevant:

- `README.md` for user-facing behavior or workflow changes
- `examples/` for changed usage patterns
- pull request description for migration and compatibility impact

## Pull request guidance

Good pull requests in this repository:

- focus on one concern
- explain why the change is needed
- call out migration or operational risk clearly
- include docs and example changes when needed

## Suggested workflow for AI-assisted edits

1. Read the relevant provider files and examples first.
2. Create a new branch for the task before making changes.
3. For independent or parallel work, create a dedicated git worktree under `.worktree/<task-name>` and use that worktree for the task.
4. Identify whether the change belongs to user-level or system-level ownership.
5. Make the smallest safe implementation change.
6. Update docs or examples if behavior changed.
7. Run formatting, vet, and tests.
8. Summarize risks, assumptions, and validation in the PR.

## Branch and worktree workflow

- Use one branch per feature, fix, or investigation.
- Prefer descriptive branch names that reflect the single concern being changed.
- Keep task-specific worktrees under `.worktree/` so the main checkout stays clean.
- When using git worktrees, create the branch and worktree together so the task starts isolated from other in-progress changes.
- Remove task worktrees when the work is merged or no longer needed.

## If you are unsure

Prefer stopping with a clear note over guessing about:

- Cassandra migration safety
- Terraform state behavior
- ownership boundaries
- backward compatibility
